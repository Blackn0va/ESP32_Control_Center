WiFi

Netze scannen, Ergebnisse landen live in den Tabellen unten. Ziel per Klick markieren/wählen, dann im Attack-Tab angreifen.

Ablauf: 1) scanap = WLAN-Netze (Access Points) suchen · 2) scansta = mit den APs verbundene Geräte (Stationen/Clients) suchen · 3) stopscan · 4) Tabelle füllt sich · 5) Zeile → select = als Ziel markieren.
Merke Marauder-Flags: -a=Access Points · -s=SSID-Namensliste · -c=Stationen/Clients.

Deauth — Geräte vom WLAN trennen

Was: Sendet „Verbindung-trennen"-Frames, sodass Geräte vom WLAN fliegen (kurzzeitig kein Internet). Ziele im WiFi-Tab anhaken — sie erscheinen hier automatisch. MAC-Modus braucht KEIN funktionierendes scanap/select.
Keine Ziele gewählt — im WiFi-Tab APs/Stationen anhaken.
Handmodus: unten in der Konsole attack -t deauth -s <Quell-MAC> -d <Ziel-MAC> tippen.

Beacon Spam — viele Fake-WLANs vortäuschen

Was: Strahlt massenhaft erfundene Netznamen aus, sodass in der WLAN-Liste der Opfer viele Fake-Netze erscheinen.

Probe Flood

Was: Sendet massenhaft Probe-Requests (Netz-Suchanfragen) für die gewählten APs — kann APs/Logs fluten. Voraussetzung: scanap + AP wählen.

🔓 WPA-Handshake / PMKID Capture (für PCAP-Cracking)

Kurz: Capture starten = deauth + Handshake-Sniff. Wo die .pcap landet, steht im Banner oben. Nur eigene / autorisierte Netze.
Kein AP-Ziel gewählt — im WiFi-Tab AP anhaken (setzt Kanal automatisch), oder Kanal unten manuell.
Bereit. Nach dem Start auf EAPOL/PMKID/handshake im Log warten (wird hier grün gemeldet).
Cracken am PC (egal woher die .pcap kommt):
hcxpcapngtool -o hash.hc22000 dein.pcap → hashcat -m 22000 hash.hc22000 wortliste.txt

Evil Portal

⚠ Nur autorisierte Tests. Braucht SD-Karte am ESP32 (ab FW v0.11). HTML nur von SD.

ESP32-S3-DevKitC-1 hat keinen SD-Slot. Für Portal SPI-microSD-Modul verdrahten (SCK/MISO/MOSI/CS, Pins im configs.h des Builds). Ohne SD: Portal nicht nutzbar — WiFi/BLE/Attack gehen aber ohne SD.
Was ist Evil Portal: Marauder macht ein WLAN auf und zeigt jedem, der sich verbindet, eine Login-Seite (Captive Portal). Eingegebene Daten werden auf SD geloggt. Zum Testen des eigenen Netzes / Awareness-Schulung.

⚡ Flash / Wartung

Kompatibel mit allen ESP32-Varianten (ESP32, S2, S3, C3, C5, C6, H2). esptool-js erkennt den Chip automatisch.

Chip: — (erst verbinden) Lib: —
A · ESP32-S3 Voll-Reparatur (eingebautes Komplett-Image)
Nur für ESP32-S3. Enthält Bootloader+Partitionen+App — repariert auch komplett leere/gebrickte S3.
Eingebaut: — — komplettes Image @0x0.
B · Universal — Marauder-App flashen (jeder Chip)
Für jeden ESP32, der schon einen Bootloader hat (Normalfall). Lädt die App deines Boards von GitHub und schreibt sie @0x10000 — Bootloader/Partitionen bleiben erhalten. Chip wird beim Flashen erkannt.
Board oben wählen (volle Liste) oder frei filtern. DevKitC-1 = multiboardS3. Unsicher? „🎯 Auto-Flash" erkennt den Chip selbst.
App flashen = Update, Board hat schon Bootloader. Voll-Reparatur = auch Bootloader+Partitionen neu (für leere/gebrickte Boards, jeder Chip: ESP32/S2/S3/C3). Download: bevorzugt lokaler Proxy (starte den Server mit python serve.py statt http.server), sonst öffentliche CORS-Proxies. „Proxy erzwingen" überspringt den Direktversuch. Klappt gar nichts: manuell (C) / flasher.marauder.gg.

C · Manuell / Erweitert (Voll-Flash, eigene Dateien)

Auto: Offsets werden nach erkanntem Chip gesetzt. Classic/S2 haben Bootloader @0x1000, alle neueren @0x0.
1. Verbinden (ROM/Download-Modus)
Fehler? BOOT halten → EN/RST tippen → BOOT los → Button.
2. Flash löschen (optional)
3. Dateien wählen
Voll-Flash: bootloader @0x0 (classic @0x1000), partitions @0x8000, app @0x10000. Nur-App: eine Datei @0x10000.
4. Flashen
Recovery (im Terminal-Modus)
Voll-Flash für Nicht-S3 (Bootloader fehlt/leer) am sichersten via flasher.marauder.gg — hat komplette Manifeste pro Board.
